UNIX环境高级编程学习笔记二

前言

        第二章UNIX标准化及实现的学习笔记

UNIX标准化

        由于Unix的盛行，制定标准是必须的事情，不然就不同厂商做的都不一样，程序的移植性就非常的差。从而有了以下三个标准：

  • ISO C
  • IEEE POSIX
  • Single UNIX Specification

    三个标准，是接口的规范，厂商根据接口规范编程具体实现

UNIX系统实现

        有了标准，厂商就得根据标准做具体的实现，于是有了三个分支

  • AT&T分支(UNIX的商用版本)
  • 加州大学伯克利分校分支
  • AT&T不断开发的UNIX研究版本

        三个分支对应了以下的版本

  • SVR4(UNIX System V Release 4)
  • 4.4BSD(Berkeley Software Distribution)
  • FreeBSD
  • Linux
  • Mac OS X,核心是Darwin，是一种Unix-like操作系统
  • Solaris，基于SVR4

        这几种都不同程度上符合POSIX标准。

限制

        为了软件移植性（比如不同系统允许打开的文件描述符最大值不同），必须在以下两方面提供限制

  • 编译时限制（短整型的最大值是多少）
  • 运行时限制（文件名有多少字符）

        编译时限制在头文件中定义，运行时限制要求进程调用一个函数获得限制值。
        比如文件名可以有多少字符取决于文件系统，因此只能在运行时调用函数获取该系统的长度限制。为了可移植性不能写在头文件里。
        下面3种限制是为了解决这个问题

  • 头文件（编译时限制）
  • 与文件或目录无关的运行时限制(sysconf函数)
  • 与文件或目录有关的运行时限制(pathconf和fpathconf函数)

        相应的各个标准的限制如下

  • ISO C限制     编译时限制都在中，定义了各种类型的上下限

  • POSIX
  • XSI限制

sysconf,pathconf,fpathconf

        这三个函数定义了运行时限制，比如同个系统但使用了不同文件系统导致的不同。

基本系统数据类型

        <sys/types.h>定义了clock_t等基本系统数据类型

标准间冲突

        有冲突时，POSIX服从ISO标准。
